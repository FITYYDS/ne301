# 为什么 Web 能运行在单片机上

## 0. 整体框图

先不要急着看代码和协议，先看一张总图。只要这张图想明白了，后面的细节就都好理解了。


这张图表达的核心意思只有一句话：

> 网页不是在 MCU 上“显示”的，而是浏览器在显示；MCU 只是把网页文件、接口数据、实时流通过网络提供出来。

很多人卡住，就是因为把“网页显示”和“网页服务”混成了一件事。实际上它们是两件完全不同的事。

## 1. 先讲最直白的结论

### 1.1 单片机上并没有跑浏览器

当你在设备的 IP 地址里看到一个登录页、配置页、预览页时，不代表 MCU 在里面跑了 Chrome、Edge、Safari。

真正发生的事情是：

1. 浏览器跑在你的电脑或手机上
2. MCU 里跑的是一个轻量级 Web 服务器
3. 这个 Web 服务器把页面文件和接口数据发给浏览器
4. 浏览器收到这些内容以后，自己完成页面显示和交互

所以，“Web 能运行在单片机上”这个说法，准确地讲应该是：

- MCU 可以运行 `Web 服务端`
- 浏览器负责 `Web 客户端显示`

### 1.2 用一个生活类比理解

你可以把整个系统想成“餐馆点餐”：

- 浏览器：顾客的手机
- Web 页面：菜单界面
- MCU 里的 Web 服务：接单员
- MCU 里的业务服务：后厨
- 驱动和硬件：锅灶、冰箱、摄像头、Flash、网卡

顾客手机上看到菜单，不代表厨房里有一部手机在显示菜单；只是厨房后面的系统把菜品信息发到了顾客手机上。

MCU 做的事和这个完全类似：

- 它不负责把按钮画出来
- 它只负责告诉浏览器：“这里有按钮，这里有数据，这里有视频流”

## 2. 你打开一个设备网页时，背后到底发生了什么

这一段最重要。只要你把一次完整访问过程想清楚，基本就不会再困惑了。

### 2.1 你在浏览器地址栏输入设备 IP

比如你输入：

```text
http://192.168.10.10
```

这时浏览器会发起一个 HTTP 请求：

```text
GET /
```

### 2.2 MCU 里的 Web 服务收到这个请求

MCU 上有一个程序一直在监听端口，比如：

- `80`
- `8080`
- `443`

监听到 `GET /` 后，它会判断：

- 这是要页面
- 不是要业务接口

于是它会去找首页资源，一般是：

- `index.html`

### 2.3 MCU 把 `index.html` 返回给浏览器

注意这里的重点：

- MCU 不会“显示”这个 HTML
- MCU 只是把 HTML 文件内容发出去

浏览器收到 HTML 后，开始自己解析页面结构。

### 2.4 浏览器继续请求 JS、CSS、图片等资源

在 `index.html` 里通常还会引用：

- `app.js`
- `style.css`
- logo 图
- 字体文件

于是浏览器又继续请求这些文件：

```text
GET /assets/app.js
GET /assets/style.css
GET /assets/logo.png
```

MCU 继续按文件路径把这些资源发给浏览器。

### 2.5 浏览器执行前端代码

等 `app.js` 下载下来后，浏览器会执行这段 JavaScript。前端代码这时才会去做真正的业务动作，例如：

- 请求设备信息
- 请求网络配置
- 查询 AI 状态
- 建立视频预览连接

比如它可能发出：

```text
GET /api/v1/device/info
POST /api/v1/system/network/wifi
GET /api/v1/work-mode/status
```

### 2.6 MCU 把这些接口映射到本地业务

这时候 MCU 不再返回静态文件了，而是：

1. 解析 URI 和 HTTP 方法
2. 找到对应 API handler
3. 调用本地业务服务
4. 业务服务再去读配置、查状态、调用驱动
5. 最终把结果封装成 JSON 返回

例如：

```json
{
  "success": true,
  "data": {
    "device_name": "NE301",
    "firmware_version": "1.0.0"
  }
}
```

### 2.7 如果是实时预览，还会建立 WebSocket

对于视频、波形、日志推送这类实时数据，HTTP 一问一答不够方便，通常会再建立一条长连接：

```text
ws://192.168.10.10:8081/stream
```

后面 MCU 就不断把实时数据推给浏览器，浏览器负责显示出来。

### 2.8 所以整件事其实很简单

```text
浏览器负责显示
MCU 负责提供内容
```

这就是“Web 在 MCU 上运行”的本质。

## 3. 为什么 MCU 明明资源很小，也能做这件事

这也是很多人的第二个疑问。

### 3.1 因为最重的工作不在 MCU 上

网页真正重的部分是：

- 页面布局
- 样式渲染
- JavaScript 运行
- 动画
- 视频显示

这些都在浏览器里完成。

MCU 做的是：

- 接收请求
- 查找资源
- 返回数据

从计算量上讲，这比“自己画完整 UI”轻很多。

### 3.2 因为 MCU 端通常是事件驱动，不是重量级服务模型

桌面或服务器 Web 通常会有很多重量级机制：

- 多进程
- 多线程池
- CGI
- 数据库
- 文件系统缓存
- 大量中间件

MCU 上一般不会这样，而是采用非常克制的模型：

- 一个 Web 线程
- 一个事件循环
- 非阻塞 socket
- 回调处理
- 固定大小缓冲区

它追求的不是“功能很全”，而是“够用、稳、占用低”。

### 3.3 因为页面不是在设备上动态生成的

MCU 里的页面，绝大多数不是设备临时拼出来的，而是开发阶段就提前准备好的：

1. 在 PC 上把前端工程构建好
2. 压缩
3. 打包
4. 烧录到 Flash
5. 运行时按文件发送

也就是说，设备只是“快递员”，不是“现场排版员”。

### 3.4 因为很多设备页面其实并不重

设备 Web 页面常见内容无非是：

- 登录页
- 参数设置页
- 状态页
- 升级页
- 日志页
- 预览页

除了预览流之外，大多数交互都是小 JSON 包，并没有你想象得那么重。

## 4. MCU 想跑 Web，到底需要哪些东西

如果你把 MCU Web 拆开来看，它至少需要 5 层。

## 4.1 硬件联网能力

设备必须先有网络入口，例如：

- 以太网
- WiFi
- 蜂窝模块
- USB 虚拟网卡

没有网络，就没有 Web。

## 4.2 TCP/IP 协议栈

设备要能理解并处理网络协议，通常至少要支持：

- IP
- ARP
- TCP
- Socket

常见实现有：

- `lwIP`
- `NetX Duo`
- `FreeRTOS+TCP`

## 4.3 HTTP / WebSocket 协议处理

这一层就是“Web 服务器”本体，它负责：

- 监听端口
- 接收请求
- 解析 HTTP
- 返回响应
- 处理 WebSocket 升级

常见库包括：

- `Mongoose`
- `lwIP httpd`
- `CivetWeb`

## 4.4 资源存储

要把页面发给浏览器，就得先把资源放在设备里。常见方式有：

- 烧在内部 Flash
- 烧在外部 QSPI Flash
- 放在文件系统中
- 放在固定地址资源镜像中

## 4.5 业务服务层

这一层是真正“接设备能力”的地方。比如：

- 获取设备信息
- 设置 WiFi
- 控制工作模式
- 获取 AI 状态
- 启动预览
- OTA 升级

如果没有这一层，Web 只是一个只能返回文件的空壳。

## 6. 单片机 Web 和 Linux Web 有什么本质区别

原理相似，但工程做法差别非常大。

| 对比项 | Linux / PC Web | MCU Web |
| --- | --- | --- |
| 资源 | 内存和存储比较充裕 | Flash 和 RAM 很紧张 |
| 运行模型 | 多进程、多线程、容器都很常见 | 事件驱动、少线程更常见 |
| 文件系统 | 非常完整 | 常常没有完整文件系统 |
| 页面资源 | 本地磁盘直接读取 | 多数是预打包资源镜像 |
| 动态内容 | 复杂模板和数据库联动常见 | 多是静态资源 + JSON API |
| 故障恢复 | 可重启进程 | 弄不好会拖垮整机 |
| TLS | 配起来相对容易 | 支持可以，但成本高 |

所以 MCU Web 的核心思路不是“搬一个 PC Web 服务器过去”，而是：

> 只实现设备真正需要的那一小部分 Web 能力。

## 7. Web 是如何被“移植”到单片机上的

这一节按工程顺序来讲。

## 7.1 第一步：先把网络打通

最先做的绝不是前端，而是：

1. 网卡或 WiFi 驱动跑起来
2. TCP/IP 栈跑起来
3. 设备获得 IP
4. 能 ping 通
5. 能建立 TCP 连接

如果连这个都没通，后面所有 Web 都无从谈起。

## 7.2 第二步：放进一个轻量 Web 服务器

有了网络以后，再接一个嵌入式 HTTP 库。这个库通常会帮你做：

- 监听端口
- 接收连接
- 解析 HTTP
- 提供回调
- 处理 WebSocket

这一步以后，设备就有能力对浏览器说：

- “我这里有一个首页”
- “我这里有一个接口”
- “我这里能推实时数据”

## 7.3 第三步：设计 URL 和接口

一般会把请求分成两类：

### 一类是静态资源

例如：

- `/`
- `/index.html`
- `/assets/app.js`
- `/assets/style.css`

### 一类是业务接口

例如：

- `/api/v1/device/info`
- `/api/v1/system/network/status`
- `/api/v1/system/ota/upload`

这样浏览器和 MCU 的职责边界就很清晰。

## 7.4 第四步：把前端页面打包进设备

这一步最像“移植”。

一个现代前端工程通常在 PC 上先构建：

```text
React / Vue / Preact 源码
    -> vite build / webpack build
    -> dist 目录
    -> 压缩
    -> 打包成资源镜像
    -> 烧录到 Flash
```

这样 MCU 运行时就不用理解前端源码，只需要：

- 根据 URL 找到资源
- 读取二进制内容
- 发给浏览器

## 7.5 第五步：把 API 接到业务服务上

这一步是真正让页面“活起来”的关键。

例如页面上有一个“切换 WiFi”按钮。浏览器点击后，通常会：

1. 发送一个 `POST /api/v1/system/network/wifi`
2. MCU 收到请求
3. API handler 解析 JSON
4. 调用网络服务层
5. 网络服务层调用驱动或模组接口
6. 返回成功或失败

所以 HTTP 只是门口的接待员，真正干活的是服务层。

## 7.6 第六步：对于实时数据，增加 WebSocket

HTTP 更适合：

- 查询状态
- 修改配置
- 上传文件

WebSocket 更适合：

- 视频预览
- 实时日志
- 实时告警
- 高频状态刷新

因此在摄像机、工控、边缘 AI 设备里，很常见的组合是：

- `HTTP` 做控制面
- `WebSocket` 做数据面

## 7.7 第七步：处理 MCU 特有问题

移植 Web 到 MCU，最容易出问题的不是“页面能不能打开”，而是下面这些工程问题：

- 内存够不够
- 上传大文件会不会爆内存
- 同时连接多个浏览器会不会卡死
- 视频流和控制请求会不会互相影响
- OTA 写 Flash 时 Web 线程怎么办
- 低功耗模式要不要保留 Web 服务
- 断电恢复如何保证资源一致性

这部分才是嵌入式 Web 真正难的地方。

```text
    |

    v
```

## 9. 结合 NE301 看它是怎么做的

现在开始把理论落到这个项目本身。

## 9.1 NE301 的 Web 不是一个概念，而是一整套组合

结合工程源码，NE301 的 Web 方案可以概括为：

```text
CMSIS-RTOS2 + lwIP + Mongoose + Flash 资源镜像 + WebSocket 视频流
```

这 5 个部分分别承担不同角色：

- `CMSIS-RTOS2`：线程、事件、同步
- `lwIP`：TCP/IP 和 socket
- `Mongoose`：HTTP / WebSocket 协议处理
- `Flash 资源镜像`：存页面资源
- `WebSocket 视频流`：实时预览

## 9.2 NE301 的前端在什么地方

前端工程在：

- `Web/`

从工程内容看，它使用了：

- `Preact`
- `TypeScript`
- `Vite`

也就是说，NE301 的页面本质上是一个现代前端工程，只不过最终部署目标不是 Linux 文件系统，而是 MCU 的 Flash。

## 9.3 NE301 的 MCU 侧 Web 服务在什么地方

主要代码在：

- `Custom/Services/Web/web_service.c`
- `Custom/Services/Web/web_server.c`
- `Custom/Services/Web/web_assets.c`
- `Custom/Services/Web/websocket_stream_server.c`

这几个文件刚好对应前面讲的几层：

- `web_service.c`：统一入口和生命周期
- `web_server.c`：HTTP 服务和 API 路由
- `web_assets.c`：静态资源解析和查找
- `websocket_stream_server.c`：实时流服务

## 10. NE301 的 Web 是怎么启动起来的

NE301 不是在 `main()` 里随便起一个 socket 就算了，它是纳入服务框架统一管理的。

在 `Custom/Services/service_init.c` 里，`web_service` 被当作一个正式服务模块注册：

- 名称：`web_service`
- 依赖：`communication_service`
- `auto_start = true`
- 低功耗下默认不是必须服务

这说明项目设计者的思路是：

1. 网络服务先起来
2. Web 再起来
3. 低功耗模式下 Web 可以不启动

这很符合产品化设备的设计习惯。

## 11. NE301 的 HTTP 服务是如何工作的

在 `Custom/Services/Web/web_server.c` 中，可以看到它使用 Mongoose 的事件驱动模型。

核心动作有三个：

### 11.1 初始化 Mongoose manager

```c
mg_mgr_init(&g_web_server.mgr);
```

这相当于先准备一个连接管理中心。

### 11.2 创建监听端口

```c
mg_http_listen(&g_web_server.mgr, listen_addr, web_server_event_handler, &g_web_server);
```

这一步相当于告诉系统：

- 在某个端口上等着浏览器来连
- 有人来了就交给 `web_server_event_handler`

### 11.3 进入循环持续 poll

```c
while (g_web_server.running) {
    mg_mgr_poll(&g_web_server.mgr, 10);
}
```

这就是嵌入式 Web 很典型的工作方式：

- 不开很多进程
- 不做很重的线程池
- 就是在一个循环里不断处理网络事件

这也解释了为什么 MCU 也能撑住：它的模型足够轻。

---

## 12. NE301 的 Web 跑在哪个线程里

`web_server.c` 里给 HTTP 服务单独创建了线程：

- 线程名：`web_server`
- 优先级：`osPriorityRealtime`
- 栈大小：`1024 * 32`

还创建了一个辅助线程：

- `web_server_ap_sleep_timer`

这说明它不是“顺便跑一下”的临时代码，而是产品级的长期服务线程。

你可以把它理解成：

- `web_server` 线程负责接请求、回页面、回接口
- `ap_sleep_timer` 线程负责辅助超时和 AP 休眠管理

---

## 13. NE301 如何区分“页面请求”和“接口请求”

这是项目里很关键的一个设计点。

在 `web_server_handle_request()` 中，请求会被分成两类：

### 13.1 静态资源请求

例如：

- `/`
- `/index.html`
- `/assets/...`

这类请求最终会走：

- `web_server_handle_static_request()`

它会去资源表里查文件，然后把文件内容通过 HTTP 发回去。

### 13.2 API 请求

项目里 API 前缀定义在 `Custom/Services/Web/web_api.h`：

```text
/api/v1
```

凡是这类 URI，会走：

- `web_server_handle_api_request()`

然后再进一步找到对应的模块 handler。

这就是典型的前后端分离做法：

- 页面文件是一类
- 业务能力是一类

---

## 14. NE301 的页面资源是怎么放进 MCU 的

这部分最值得你重点看，因为这正是“Web 如何移植到 MCU”最实际的一步。

## 14.1 页面先在 PC 上构建

在 `Web/package.json` 里有一条关键命令：

```text
build:bin = tsc -b && vite build && node static_to_c_converter.cjs -i ./dist -o ./firmware_assets && node ./scripts/pack_web_assets.cjs
```

这条命令说明整个链路是：

1. `tsc -b`
   - 先把 TypeScript 编译好
2. `vite build`
   - 生成前端发布产物
3. `static_to_c_converter.cjs`
   - 把产物打包成 MCU 可部署资源
4. `pack_web_assets.cjs`
   - 进一步打成 OTA 包

## 14.2 资源不是散文件，而是一个镜像包

`Web/static_to_c_converter.cjs` 里会做几件事：

- 收集 `dist/` 下所有静态文件
- 对适合压缩的内容做 gzip
- 生成资源头和索引
- 生成 `web-assets.bin`

这个资源包里有：

- magic：`WEBASSETS`
- 版本号
- 文件数
- 总大小
- 压缩后大小
- 时间戳
- 索引表
- 文件数据区

这说明它采用的是“资源镜像”思路，而不是“把一堆网页散着拷进去”。

## 14.3 资源烧录到固定 Flash 地址

在 `Custom/Services/Web/web_service.c` 中可以看到：

```c
#define WEB_ASSETS_FLASH_ADDRESS 0x70400000
```

并且初始化时会调用：

```c
web_asset_adapter_init((const uint8_t*)WEB_ASSETS_FLASH_ADDRESS);
```

这说明页面资源最终被烧到了：

- `0x70400000`

运行时直接从这个地址解析资源表。

## 14.4 MCU 运行时如何查文件

`Custom/Services/Web/web_assets.c` 里做的事情是：

1. 从 Flash 指针位置读取头部
2. 校验 magic 是否是 `WEBASSETS`
3. 解析文件索引
4. 为每个资源建立 `path -> data` 映射
5. 判断资源是否 gzip 压缩

然后当浏览器请求：

```text
GET /assets/app.js
```

MCU 端就会去资源表里找到对应数据块，再把它发出去。

所以它根本不需要一个传统 PC 意义上的完整文件系统。

---

## 15. NE301 的 API 为什么能驱动设备业务

因为 Web 层后面挂的是服务层，而不是直接操作硬件。

你可以把 NE301 的 API 调用路径理解成：

```text
Browser
  -> /api/v1/...
  -> web_server.c
  -> API module
  -> system/device/network/ota/ai service
  -> driver or core module
  -> JSON response
```

这条路径非常关键。它代表：

- Web 只是一个接入层
- 真正的业务在 service 层
- 真正的硬件操作在 driver / core 层

这样做的好处是：

1. 结构清晰
2. 好维护
3. 将来换成 MQTT 或串口协议，也能复用业务层

## 16. NE301 的视频为什么也能在网页里预览

很多人会在这里再次产生误解，以为 MCU 在渲染网页视频。

其实不是。

## 16.1 MCU 负责的是“发视频数据”

在 `Custom/Services/Web/websocket_stream_server.c` 中：

- 默认端口是 `8081`
- 路径是 `/stream`
- 匹配到后调用 `mg_ws_upgrade()`
- 实际发送用 `mg_ws_send()`

它还会订阅：

- `video_hub`

这说明 MCU 做的是：

- 从视频链路拿到编码后的数据
- 通过 WebSocket 发给浏览器

## 16.2 浏览器负责的是“把视频画出来”

前端代码在：

- `Web/src/lib/MSE`

并且工具函数里明确默认拼接：

```text
ws://host:8081/stream
```

也就是说，浏览器连上这条 WebSocket 后，收到的是视频数据流，然后由浏览器侧代码去喂给播放器。

所以职责边界仍然很清楚：

- MCU 发送视频数据
- 浏览器显示视频画面

---

## 17. 用 NE301 这套实现反过来总结原理

通过这个项目，你其实已经能看到一个很完整的结论：

### 17.1 Web 能运行在单片机上，不是因为单片机能渲染网页

而是因为：

- 浏览器帮它承担了渲染工作
- MCU 只承担服务端工作

### 17.2 Web 能被移植到单片机上，不是把前端源码搬进去

而是：

- 先在 PC 上构建前端
- 再打包成资源镜像
- 最后在 MCU 上通过 HTTP 把资源送给浏览器

### 17.3 实时功能不是靠轮询页面死刷

而是：

- 控制类功能走 HTTP API
- 实时数据走 WebSocket

这是一种非常典型、也非常合理的嵌入式 Web 设计。

## 18. 如果你自己要做一套 MCU Web，推荐的落地顺序

很多人一上来就想做漂亮界面，但工程上更稳的顺序是下面这样。

## 18.1 先做到最小闭环

第一阶段目标只要这 4 件事：

1. 设备拿到 IP
2. 浏览器能访问设备
3. `GET /` 能返回一个最简单 HTML
4. `GET /api/ping` 能返回 JSON

只要这四步没打通，就还没有真正完成 Web 移植。

## 18.2 再把资源打包机制做起来

第二阶段做：

1. 把多个静态文件整理出来
2. 建资源索引
3. 支持 gzip
4. 放到 Flash 固定地址

这样页面加载机制就稳定了。

## 18.3 再把业务服务接进去

第三阶段再做：

- 设备信息
- 网络配置
- 参数保存
- 日志导出
- 固件升级

这一步应该强调“服务层隔离”，不要让 HTTP handler 到处直接碰驱动。

## 18.4 最后再上实时流和复杂前端

第四阶段再考虑：

- WebSocket
- 视频预览
- 前端框架
- 多页面
- 国际化
- 文件上传

这才是更稳的工程顺序。

## 19. 一个推荐的最小架构模板

如果从零开始做一套 MCU Web，我会建议先收敛成下面这个模板：

```text
web_task
  |
  +-- network already ready
  +-- http server init
  +-- asset table init
  +-- api route register
  `-- event loop

asset store
  |
  +-- index.html
  +-- app.js
  `-- style.css

api router
  |
  +-- GET /api/ping
  +-- GET /api/info
  `-- POST /api/config

services
  |
  +-- device_service_get_info()
  +-- config_service_get()
  `-- config_service_set()
```

这套结构很小，但已经具备完整 Web 的核心骨架。

## 20. 最后一句最通俗的总结

如果要把这件事用最通俗的一句话讲给非软件背景的人听，可以这么说：

> 单片机并不是在里面打开了一个网页，而是像一个小型网站服务器一样，把网页文件和设备数据发给你电脑上的浏览器，所以你才能在浏览器里看到它的管理界面。

而 NE301 这套工程，就是把这件事做成了一个完整产品：

- 前端页面先在 PC 上编译
- 页面资源打包进 Flash
- STM32N6 运行轻量 Web 服务
- 接口调用本地业务服务
- 视频通过 WebSocket 推送
- 最终由浏览器负责显示整个界面

这就是“为什么 Web 能运行在单片机上”，以及“Web 是如何被移植到单片机上的”完整答案。
